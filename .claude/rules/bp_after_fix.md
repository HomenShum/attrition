# Re-run Workflows After Code Changes

After modifying code that was originally produced by a captured workflow, re-run the workflow's judge session to verify the replay still works.

## When to activate

- After editing files that appear in a captured workflow's event stream
- After refactoring code that was generated by a distilled workflow replay
- When the user says "verify replay", "check workflow", or "does this still replay?"
- Before merging a PR that touches workflow-captured code paths

## Protocol

1. List workflows: `bp workflows`
2. Find workflows that touch the modified files (check event streams for matching paths)
3. Start a judge session: `bp judge <workflow-id> --replay-model <current-model>`
4. Use `bp.judge.event` MCP tool to feed actual events during re-execution
5. Finalize with `bp.judge.verdict` to get pass/fail

## Interpreting verdicts

- **correct**: Replay matches the original workflow perfectly. Safe to ship.
- **partial** (score > 0.8): Minor divergences. Review the divergence list but likely OK.
- **partial** (score < 0.8): Significant divergences. The code changes may have broken the workflow.
- **escalate**: Too many major divergences. Human review required before shipping.
- **failed**: Critical divergences detected. The replay produced incorrect results. Investigate before merging.

## After a failed verdict

1. Check which events diverged (the verdict includes a divergence list)
2. Determine if the divergence is expected (intentional refactor) or a regression
3. If expected: re-capture the workflow with `bp capture` to establish a new baseline
4. If regression: fix the code and re-run the judge session
