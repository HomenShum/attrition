Incident Summary: Stripe Webhook 504 Gateway Timeouts

Incident Overview: Between 2026-04-19 06:30-07:00 PDT, our payments.stripe.webhook service experienced 504 Gateway Timeouts, resulting in 420 missed customer payment events.

Root Cause: The 504 errors were caused by synchronous database lock contention during high-volume webhook processing. As identified in ISS-PAY-033, the service attempted to perform heavy write operations (updating user subscription states) directly within the request-response cycle. This exceeded the load balancer's timeout threshold, preventing the required 2xx acknowledgment to Stripe.

---

Recovery and Remediation Plan

To restore data consistency and prevent recurrence, please execute the following steps:

#### 1. Immediate Data Recovery (Reconciliation) Per POL-OPS-011, do not manually insert records into the database. Instead, follow this recovery protocol:

• Identify Missed Events: Navigate to the Stripe Dashboard > Developers > Webhooks. Filter for the impact window of 2026-04-19 06:30-07:00 PDT to identify the 420 events that returned a 504 status.
• Replay Events: Use the Stripe Dashboard or the Stripe CLI (stripe events resend <event_id>) to trigger a replay of these events.
• Verify State: Ensure your webhook endpoint is now configured to process these events asynchronously. Use the Stripe API (GET /v1/events/<event_id>) to verify that the replayed events correctly update your local database records.

#### 2. Immediate Service Stability

• Decouple Processing: Refactor the webhook receiver to immediately return a 2xx status code upon receipt. All business logic (database writes/subscription updates) must be offloaded to an asynchronous background worker (e.g., Redis/Sidekiq) to ensure the webhook remains responsive.
• Resource Throttling: Implement concurrency limits on your background worker pool to prevent future database lock contention.

#### 3. Long-term Prevention

• Infrastructure Optimization: Review ISS-PAY-033 to optimize database transaction isolation levels for subscription updates.
• Monitoring: Implement automated alerts for 5xx spikes on the webhook endpoint to trigger circuit breaking before the 30-minute threshold is reached.
• Compliance: Ensure all future webhook handling adheres to POL-OPS-011, prioritizing rapid 2xx responses to maintain Stripe connection health.
