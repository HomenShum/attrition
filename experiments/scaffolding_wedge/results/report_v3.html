Method

Dataset: Real MMLU-Pro questions fetched from HuggingFace (TIGER-Lab/MMLU-Pro). Graduate-level, 10 multiple-choice options, deterministic correct answer. No judge subjectivity — correctness is letter-match.

Sample: 15 questions across 3 domains (business, law, psychology), 5 each.

Capability ladder:

  • gemini-2.5-flash-lite (weakest, cheapest, $0.10/$0.40 per M)
  • gemini-3.1-flash-lite-preview (mid, $0.075/$0.30 per M)
  • gemini-3.1-pro-preview (strongest, $1.25/$5.00 per M)
  • Weak + domain-distilled reasoning skill (hand-crafted checklist per domain)

Note on older Gemini: Gemini 1.0 and 1.5 are fully retired. Gemini 2.0 retires June 2026. 2.5-flash-lite is the weakest currently available tier.

Key Findings (brutally honest)

Finding 1: The skill DOES transfer reasoning.
Weak + skill scored 60.0% vs weak alone at 46.7% — an uplift of +13.3pp. On 4 questions the skill rescued a wrong weak-model answer. On 2 questions the skill HURT performance (weak alone got it right, weak+skill got it wrong). Net positive: distillation is not placebo; it moves the needle.
Finding 2: But it doesn't close the gap to a stronger model.
Weak + skill (60.0%) still trails strong (86.7%) by 26.7pp. Quality retention is only 69.2%, below the 80% threshold needed for the wedge to be commercially viable on this benchmark. Some questions require capability the weak model doesn't have, and no checklist fixes that.
Finding 3: The skill prompt eats the cost savings.
The distilled skill adds ~700-900 input tokens. Cost fraction is 95.8% — weak+skill is nearly as expensive as strong alone on this benchmark. Naive scaffolding injection doesn't work economically unless the skill can be cached, compressed, or shared across many queries.
Finding 4: The REAL winner is the MID tier.
gemini-3.1-flash-lite-preview scored 80.0% — only 6.7pp behind strong — at total cost of $0.000318 (6.0% of strong). For many workloads, the right optimization isn't distillation — it's picking the right model tier in the first place. This reframes attrition's value: not "use a cheaper model with a skill" but "measure capability vs cost per query type and route accordingly."
Finding 5: Strong is not always better.
On 1 questions, strong got it WRONG while a smaller model got it right. Blindly defaulting to Pro wastes money without guaranteeing quality.

First-Principles Conclusion (V2 vs V3 cross-check)

Why V2 said "WEDGE CONFIRMED DOMINANT" and V3 says "PARTIAL"

V2 used FloorAI synthetic queries judged by a Pro judge. Pro-as-judge biases toward responses that look structured and policy-grounded — which is exactly what the distilled skill produces. The judge rewarded form over correctness.

V3 uses real MMLU-Pro with DETERMINISTIC correct answers. No judge bias. Correctness is letter-match. And the signal is subtler: skills help, but not enough to close the capability gap on hard questions.

What's actually true after both experiments

  • Skills transfer SOME reasoning: +13.3pp on weak model, rescue 4 wrong answers.
  • Skills don't close the capability gap: weak+skill 60.0% vs strong 86.7%. The cheapest model can't reason like the best one.
  • Prompt overhead kills economics: skill tokens make cheap+skill as expensive as strong on this benchmark.
  • Model tier selection beats skill injection: mid tier gives 92% of strong's accuracy at 6% of the cost. No skill needed.
  • Strong is not a safe default: strong was wrong on 1 questions a smaller model got right.

Honest commercial implication

attrition's viable wedge is NOT distillation-as-a-service. It is per-query capability routing backed by measurement:

  1. Measure which queries actually benefit from Pro capability (MMLU-Pro-style ground truth).
  2. Route the rest to mid-tier or weak+skill based on measured outcomes, not guesses.
  3. Only apply scaffolding where the data shows it recovers >X% of capability gap.

That's a routing + measurement engine, not a distillation engine. Measurement is what nobody else does against real ground truth. That's the defensible wedge.

Next experiment (V4)

  1. 100+ questions across 5+ domains for statistical confidence.
  2. Per-query classifier: train on "which tier succeeds on this query?" using MMLU-Pro labels.
  3. Skill caching: move skill into cacheable system prompt, test whether it recovers the cost advantage.
  4. Hard-subset test: split MMLU-Pro into easy (mid gets it) vs hard (only strong gets it) and test if skill helps MORE on hard questions.
